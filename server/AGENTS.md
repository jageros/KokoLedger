# Server 工程规则

本文件约束 `server/` 目录内的 Go + go-zero 服务端开发。后续 Agent 在本目录工作时，应优先遵守这里的规则。

## 1. 不要修改生成代码

- 禁止修改 `internal/model/*_gen.go`。
- 禁止修改 goctl 生成的路由注册文件：`internal/handler/routes.go`。
- 禁止直接修改 `internal/types/*.go`，类型变更应从 `doc/api/*.api` 生成。
- 禁止直接修改 `doc/swagger/*`，Swagger 文档应由 `make api` 生成。
- 任何带有 `Code generated by goctl. DO NOT EDIT.` 的文件都视为生成文件。
- `Code scaffolded by goctl. Safe to edit.` 的文件不是生成禁区，但仍应按分层规则谨慎修改。

## 2. 不要绕过生成源头

- HTTP API、路由、handler 参数、请求/响应 types 需要变更时，先改 `doc/api/` 下的 `.api` 文件，再在 `server/` 目录执行 `make api`。
- 当前服务入口 API 文件是 `doc/api/koko.api`；后续若服务名变化，仍以 `doc/api/*.api` 和 `make api` 作为源头。
- 数据库结构需要变更时，先写 `doc/migrations/` 下的 goose SQL 迁移，再执行 `make migrate` 验证。
- 数据库模型需要变更时，先完成迁移并连到目标数据库，再执行 `make model` 或 `make model-all`。
- 单表模型生成使用 `make model table=<table_name>`；批量模型生成使用 `make model-all`，表清单维护在 `Makefile` 的 `tablesWithCache` / `tablesWithoutCache`。
- 模型自定义查询或扩展方法写到同名非 `_gen.go` 文件，例如 `internal/model/auth_sessions_model.go`。

## 3. 不要回退用户已有改动

- 开始前查看 `git status --short`。
- 发现无关脏文件、未跟踪文件或 IDE 文件时保持原样。
- 只修改完成当前任务必需的文件。
- 不要使用 `git reset --hard`、`git checkout -- <path>` 等会丢失用户改动的命令，除非用户明确要求。

## 4. 业务逻辑优先写在 logic 层

- handler 只做请求解析、创建 logic、调用 logic、返回响应。
- 业务判断、权限校验、数据库读写、第三方调用、缓存更新和错误包装写在 `internal/logic/`。
- 重复用到的公共方法可提取为同包 helper；非多次调用的方法写在 `internal/logic/` 对应的 logic 文件即可。
- 新增长期依赖、数据库连接、客户端、中间件时，集中放到 `internal/svc/service_context.go` 和相关配置结构。
- 权限规则不要只依赖前端隐藏入口，服务端 logic 层也必须校验。

## 5. 数据库与模型规则

- goose 迁移文件放在 `doc/migrations/`。
- 迁移必须包含 `-- +goose Up` 和 `-- +goose Down`。
- PostgreSQL 表字段使用 snake_case；API/Go types 可按 go-zero 生成结果对外保持 camelCase JSON。
- 金额字段使用 minor unit 的整数类型，不用浮点数存储金额。
- 软删除、归档、状态流转等业务字段应在迁移和 logic 中保持一致。
- 不要手写或修改 `internal/model/*_gen.go`；模型查询扩展写在对应的 `internal/model/*_model.go`。

## 6. 修改完成后必须验证

- Go 代码修改后至少运行相关包测试。
- 提交或交付前优先在 `server/` 目录运行 `go test ./...`。
- API 文档变更至少运行 `goctl api validate --api doc/api/koko.api`；需要刷新生成代码时运行 `make api`。
- 数据库迁移变更应使用临时 PostgreSQL 执行 goose `up` / `down` 验证，或至少说明无法验证的原因。
- 文档-only 修改可以不跑全量测试，但要用 `git diff` 或文本检索核对内容。
